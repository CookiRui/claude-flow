# Git Workflow Rules

> Supplements the constitution with details it can't cover. If derivable from the constitution, delete it.

## Rule 1: Commit message format (per Constitution §{N})

All commit messages must follow the `type(scope): description` convention. Allowed types: `feat`, `fix`, `refactor`, `docs`, `chore`, `test`, `style`, `perf`.

```text
// ✅ Correct
feat(auth): add OAuth2 login support
fix(api): handle null response from external service
docs(readme): update installation instructions

// ❌ Wrong
fixed stuff
WIP
update
```

## Rule 2: Branch naming (per Constitution §{N})

Branch names must follow the `{branch-prefix}/{feature-name}` pattern. Common prefixes: `feat`, `fix`, `chore`, `release`, `hotfix`.

```text
// ✅ Correct
feat/{feature-name}
fix/{issue-or-bug-name}
chore/{task-name}

// ❌ Wrong
my-branch
test123
{developer-name}-branch
```

**Exceptions:** {exception-scenarios — e.g., `main`, `master`, `develop` are protected branch names and do not follow this pattern}

## Rule 3: Atomic commits (per Constitution §{N})

Each commit must serve a single, well-defined purpose. Do not bundle unrelated changes into one commit.

```text
// ✅ Correct
— One commit adds a feature
— One commit fixes a bug
— One commit updates tests for that bug fix

// ❌ Wrong
— One commit that adds a feature, fixes two bugs, and updates docs
```

## Self-check Checklist

- [ ] Does every commit message follow `type(scope): description` with an allowed type?
- [ ] Does every branch name follow `{branch-prefix}/{feature-name}`?
- [ ] Does each commit contain only one logical change (no unrelated files bundled together)?
