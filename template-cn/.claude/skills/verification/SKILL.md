# Verification — 完成前检查

## 强制声明

**在宣布任何功能或 Bug 修复"完成"之前，此检查清单必须通过。** 不接受"差不多了"、"主流程能跑"或"小问题以后再修"。如果任何检查项失败，任务状态是"进行中"，而非"完成"。

## 验证检查清单

按顺序检查各项。每项必须明确标注 PASS / FAIL / N/A：

### 1. 功能完整性

- [ ] 需求/计划中的所有功能点已实现
- [ ] 提交代码中无遗留的 TODO/FIXME/HACK 注释（提交前解决或转为 Issue）
- [ ] 边界用例已处理（空值、极端输入、并发场景）

### 2. 测试验证

- [ ] 所有现有测试通过（零失败，零跳过）
- [ ] 新功能有对应的测试覆盖
- [ ] 手动验证了主流程 + 至少 1 个边界用例

### 3. 宪法合规

- [ ] 逐条对照 `constitution.md` 检查，无违规
- [ ] 技术栈选择符合宪法要求（异步框架、日志、通信等）
- [ ] 热路径无性能违规（如适用）

### 4. 代码质量

- [ ] 无编译器警告（或确认与本次变更无关）
- [ ] 新代码遵循 `rules/coding-style.md`
- [ ] 未引入不必要的依赖

### 5. 影响范围

- [ ] 变更不影响无关模块（检查 import 变更）
- [ ] 配置文件变更已记录
- [ ] 如修改了公共接口 -> 调用方已同步更新

## 失败处理

```
检查项 FAIL？
├── 功能缺失 -> 实现，重新运行验证
├── 测试失败 -> 修复代码或测试，绝不跳过/删除失败的测试
├── 宪法违规 -> 立即修复，"已经写完了"不是借口
└── 范围超出预期 -> 通知用户，评估是否需要回滚
```

## 输出格式

验证完成后，输出摘要：

```markdown
## 验证结果

| 检查项       | 结果   | 备注          |
|-------------|--------|---------------|
| 功能完整性   | PASS   |               |
| 测试验证     | PASS   | 新增 3 个测试  |
| 宪法合规     | PASS   |               |
| 代码质量     | PASS   |               |
| 影响范围     | PASS   |               |

✅ 验证通过。可以提交。
```
