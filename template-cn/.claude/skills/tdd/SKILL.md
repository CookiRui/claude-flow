# TDD — 测试驱动开发

## 强制声明

本 Skill 适用于所有新功能开发和重构任务。**这不是建议——而是强制要求。** 唯一例外：纯配置/资源/文档变更。

## RED-GREEN-REFACTOR 循环

每个功能点必须遵循以下流程。**不允许跳过任何步骤**：

| 阶段 | 步骤 |
|------|------|
| **RED** — 先写失败的测试 | 1. 根据需求编写测试（快乐路径 + 至少1个边界用例） |
| | 2. 运行测试 → 确认 RED |
| | 3. 如果测试立即通过 → 测试有缺陷，需要重新考虑 |
| **GREEN** — 最小化实现 | 1. 编写最少代码使所有测试通过 |
| | 2. 不加未测试的功能，不做过早优化 |
| **REFACTOR** — 在绿灯状态下改进 | 1. 消除重复，改善命名 |
| | 2. 每步之后运行测试 → 必须保持 GREEN |
| | 3. 不得改变外部行为 |

## 粒度控制

- 每个循环覆盖**一个行为点**，而非整个功能
- 一个功能 = 多个循环，每个循环 ≤ 5 分钟
- 如果一个循环超过 10 分钟 → 需要进一步拆分

## 反模式

1. **先写实现，后补测试** → 测试变成橡皮图章，而非设计驱动
2. **测试只覆盖快乐路径** → 边界用例在生产环境中爆炸
3. **在 REFACTOR 阶段添加功能** → 无法隔离测试失败的原因
4. **跳过 RED 确认** → 你不知道测试是否真正测试了你想验证的内容
