# /bug-fix

## 阶段 1：诊断（不写代码）

1. 理解 Bug 症状（用户描述 + 日志 + 复现步骤）。如果信息不足，使用 AskUserQuestion
2. 定位相关代码（搜索 → 阅读 → 理解调用链 → 加载相关 Skills）
3. 输出根因分析：
   - **症状**：用户看到的现象
   - **直接原因**：哪段代码导致的
   - **根本原因**：为什么（知识缺口 / 模式违规 / 缺少检查 / 时序 / 数据 / 配置 / 第三方）
   - **影响范围**：其他地方是否存在类似问题
   - **修复方案**：最小变更 — 改什么以及为什么
   - **是否需要固化？**：是否需要添加规则 / 更新技能 / 写入记忆
4. **必须通过 AskUserQuestion 等待用户确认 — 绝不跳过此步骤**

## 阶段 2：修复（用户确认后）

1. 修改前先阅读，最小变更原则
2. **先写回归测试**（RED）— 用测试复现 Bug，确认测试失败
3. 修复代码（GREEN）— 让回归测试通过
4. 宪法合规审计

## 阶段 3：完成前验证

执行完整的 `verification` 技能清单：
- 所有已有测试 + 新回归测试通过
- 逐条对照宪法检查合规性
- 确认影响范围（其他地方是否存在类似问题？）

## 阶段 4：固化（按需）

基于阶段 1 的评估：
- 添加规则 → `.claude/rules/`
- 更新技能 → `.claude/skills/`
- 写入记忆 → `MEMORY.md`
- 输出修复摘要（Bug → 根因 → 修复 → 回归测试 → 预防措施）

## 禁止操作

- 不要在不理解 Bug 的情况下修改代码
- 不要跳过用户确认
- 不要跳过回归测试
- 不要趁机重构 Bug 周围的代码
- 不要在未确认的情况下删除"看起来没用"的代码
- 不要在所有验证检查通过前宣布"修复完成"
