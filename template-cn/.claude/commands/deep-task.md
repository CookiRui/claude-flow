# /deep-task

自主执行引擎。对任务进行复杂度分类，分解为 DAG，使用 Agent 工具并行执行，三级验证，并捕获学习记录。

## 阶段 0：复杂度分类

在做任何事之前先对目标进行分类：

| 问题 | 是 | 否 |
|------|----|----|
| 只涉及 1 个文件，无依赖，可自动验证？ | → **S** | ↓ |
| 2-5 个文件，明确的验收标准？ | → **M** | ↓ |
| 跨模块或需要架构决策？ | → **L** | ↓ |
| 需要产品/战略判断？ | → **XL** | 使用 L |

输出你的分类和推理过程。

### S — 快速路径

1. 直接执行
2. L1 自检
3. `git commit -m "checkpoint: {description}"`
4. **完成。到此为止。**

### M — 标准路径

1. 分解为 2-5 个线性子任务，每个都有验收标准
2. 顺序执行：
   - 执行子任务
   - `git commit -m "checkpoint: {task}"`
   - L1 自检
3. 对关键路径进行 L2 验证（如有）
4. **完成。到此为止。**

### L/XL — 完整引擎（继续到阶段 1）

## 阶段 1：目标审查（不做代码变更）

1. **检查已有 WIP**：读取 `.claude-flow/wip.md`。如果存在且匹配此目标：
   - 从 WIP 文件读取 `saved_at_commit` 字段
   - 与当前 `git rev-parse HEAD` 比较
   - **匹配** → 可安全从阶段 3 恢复，使用已有 DAG
   - **不匹配** → 警告用户："代码自 WIP 保存后已变更（保存于 {hash}，当前 {HEAD}）。继续恢复 / 重新开始？"通过 `AskUserQuestion`

2. **目标清晰度关卡**（L/XL 任务足够复杂，值得此检查）：
   - 范围是否清晰（包含/排除什么）？
   - 能否确定何时算"完成"？
   - 是否有模糊的术语或需求？

   | 置信度 | 操作 |
   |--------|------|
   | ≥ 0.8 | 继续 |
   | 0.5–0.8 | 陈述假设，通知用户，继续 |
   | < 0.5 | `AskUserQuestion` 提出问题 + 默认假设。**等待回复。** |

3. **评估可行性**：
   - 目标是否物理上可实现？
   - 是否有隐藏的权衡？如有 → 通知用户。
   - 置信度 < 0.3 → 展示选项，让用户决定方向。

4. **构建上下文**：
   - 阅读宪法（`.claude/constitution.md`）
   - 如果可用，运行 `python scripts/repo-map.py --format md --no-refs`（生成代码地图）
   - 加载相关 Skills

5. **阅读历史学习记录**：如果存在则读取 `.claude-flow/learnings/INDEX.md`。
   - 识别与此任务相关的领域，只读取那些领域文件（不是全部）
   - 提取与此任务相关的陷阱，作为阶段 2 的约束添加（"避免 X，因为之前失败过"）
   - 如果相同类型的前置任务成功过，将其策略作为默认方案复用
   - **清理**（如果任何领域文件超过 20 条或 last_pruned 超过 30 天则执行）：
     - 移除分数 ≤ 1 的条目
     - 合并共享相同陷阱的分数 2 条目为单个整合条目
     - 更新文件头中的 `entry_count` 和 `last_pruned`
     - 更新 `INDEX.md`

6. **规划模型路由** — 基于每个任务的复杂度，而非类别：

   | 估计复杂度 | 标准 | 模型 |
   |-----------|------|------|
   | 1（简单） | 单文件编辑，机械变更，无逻辑决策 | `haiku` |
   | 2（低） | 多文件编辑，模式清晰，简单搜索+判断 | `haiku` |
   | 3（中） | 逻辑变更，测试编写，需要推理的分析 | `sonnet` |
   | 4（高） | 跨模块重构，API 设计，复杂调试 | `sonnet` |
   | 5（架构级） | 系统级决策，权衡评估 | 保持在主上下文中 |

   为阶段 2 中的每个子任务分配复杂度分数（1-5）。分数自动驱动模型选择 — 不要按任务类型硬编码模型。

7. 输出：精炼的目标陈述 + 可行性评估 + 学习约束。
   - **仅 XL**：`AskUserQuestion` — 必须等待用户确认后才能进入阶段 2。

## 阶段 2：DAG 分解

将目标分解为子任务的 DAG（有向无环图）。

每个子任务必须包含：
- **ID**：短标识符（如 `T1`、`T2a`）
- **描述**：做什么
- **验收标准**：机器可验证的条件（测试通过 / 编译成功 / 行为可观察）
- **依赖**：必须先完成的任务 ID 列表
- **复杂度**：1-5 分（按阶段 1 第 6 步驱动模型选择）
- **文件**：要创建/修改的文件

格式：
```
- T1: 搭建数据模型 → `src/models.py` | 完成: 类型定义，导入正常 | 依赖: 无 | C: 1
- T2: 实现核心逻辑 → `src/service.py` | 完成: 单元测试通过 | 依赖: T1 | C: 3
- T3: 添加 API 端点 → `src/routes.py` | 完成: 端点响应正常 | 依赖: T2 | C: 2
- T4: 集成测试 → `tests/` | 完成: 所有测试通过 | 依赖: T2, T3 | C: 3
```

模型由复杂度推导：C:1-2 → haiku，C:3-4 → sonnet，C:5 → 主上下文。

### 分解前置检查

继续之前，验证：
- [ ] **覆盖度**：所有子任务完成 = 原始目标达成？
- [ ] **独立性**：任务间边界清晰？
- [ ] **可验证**：每个验收标准都能自动检查？
- [ ] **粒度**：每个任务 ≤ 5 分钟？如果不是，继续拆分。
- [ ] **依赖**：DAG 是无环的且顺序正确？

**XL 任务**：通过 `AskUserQuestion` 展示 DAG，必须等待确认。

### 预算关卡

执行前，从 DAG 估算总成本：

```
对每个任务：
  base = 8K 输入 + 2K 输出 tokens
  file_tokens = (任务文件的估计行数) × 4 tokens/行
  task_tokens = base + file_tokens

  model_cost = {
    haiku:  (task_tokens.in × $0.25 + task_tokens.out × $1.25) / 1M,
    sonnet: (task_tokens.in × $3 + task_tokens.out × $15) / 1M,
    main:   (task_tokens.in × $15 + task_tokens.out × $75) / 1M
  }[由复杂度推导的模型]

estimated_total = sum(所有任务成本) × 1.5   # 1.5 倍缓冲，用于 L2 审查轮次
```

**决策规则**：
- estimated_total ≤ $0.50 → 静默执行
- $0.50 < estimated_total ≤ $3.00 → 通知用户估算值，继续执行
- estimated_total > $3.00 → `AskUserQuestion`："预估成本：~${total}，{N} 个任务。继续 / 缩小范围 / 取消？"

这是粗略估算 — 标注为近似值。目的是在执行前捕获意外昂贵的 DAG，而非精确记账。

## 阶段 3：并行执行循环

```
while 存在就绪任务:
    1. 获取就绪任务（所有依赖已满足）
    2. 按冲突分组（涉及相同文件的任务 = 冲突）
    3. 无冲突 → 在一条消息中通过多个 Agent() 调用并行启动
    4. 有冲突 → 顺序执行
    5. 每批完成后：
       a. 验证已完成的任务（L1）
       b. 对之前完成的任务运行回归检查
       c. git commit -m "checkpoint: {task-id} {description}"
       d. 如果回归失败 → git revert → 带约束重新规划
       e. 更新 DAG 状态
```

### Agent 调用格式

对每个子任务，从复杂度分数推导模型并启动：
```
# 模型选择：C:1-2 → haiku，C:3-4 → sonnet，C:5 → 主上下文（不用 Agent）
Agent(
  model="{由复杂度推导的模型}",
  prompt="执行此任务：\n{task.description}\n\n验收标准：{task.criteria}\n\n要修改的文件：{task.files}\n\n重要：\n- 先阅读宪法（.claude/constitution.md）\n- 如果涉及功能代码，遵循 TDD\n- 只修改指定文件\n- 完成前验证验收标准"
)
```

在**一条消息中启动多个 Agent 调用**以实现并行执行。

### 防循环规则

- 相同策略失败两次 → 必须切换方案
- 一个任务连续 3 次失败 → 升级：
  - 置信度 0.3-0.5 → `AskUserQuestion` 提供 2-3 个选项
  - 置信度 < 0.3 → 保存 WIP，完整交接

### 失败挽救

当子任务失败时，丢弃前先考虑：
- 排除了哪些假设？
- 哪些代码/数据可复用？
- 这如何缩小了解决方案空间？

在 DAG 状态中记录挽救信息，供后续尝试使用。

### Agent 间信号通道

当并行 Agent 发现影响其他任务的信息时，使用 `.claude-flow/signals/` 进行轻量级通信：

```
.claude-flow/signals/
  {source-task}_to_{target-task}.md   # 例如 T1_to_T3.md
```

**发送信号**（由执行中的 Agent）：
- 仅在发现**改变任务间契约**时写入信号（如 API 签名变更、schema 字段重命名、新增依赖）
- 信号格式：一行摘要 + 受影响文件 + 下游任务应如何调整
- 不要为代码风格偏好或非阻塞性观察发送信号

**读取信号**（由主执行循环）：
- 每批并行任务完成后，检查 `.claude-flow/signals/` 中是否有新文件
- 如果信号目标是尚未开始的任务，将信号内容注入该任务的 Agent prompt
- 如果信号目标是已完成的任务，标记需在 L1 中重新验证
- 所有任务完成后清理信号

这是一个**尽力而为**的协议 — 不写信号的 Agent 不算违规。价值在于尽早捕获接口破坏性变更，而非在 L2 中才发现。

## 阶段 4：三级验证

### 验证级别选择器

运行验证前，评估**变更影响**以避免在低风险变更上浪费预算：

| 变更影响 | 变更的文件 | 验证级别 |
|----------|-----------|----------|
| 仅文档/注释 | `*.md`、注释 | 仅 L1 — 跳过 L2 和 L3 |
| 仅测试 | 仅测试文件 | 仅 L1 — 测试自我验证 |
| 配置/基础设施 | CI、设置、脚本 | L1 + L3（跳过 L2，直接运行完整测试套件） |
| 核心逻辑（M 任务） | 实现文件 | L1 + L2-lite（1 轮，不循环）+ L3 |
| 核心逻辑（L/XL 任务） | 实现文件 | L1 + L2-full（最多 3 轮）+ L3 |

在**阶段 3 完成后**应用此选择器。运行 `git diff --stat` 来分类实际变更属于哪个类别，不论原始任务复杂度。

### L1 — 自检（阶段 3 中，每个子任务）
每个 Agent 验证自己的验收标准。已在执行循环中完成。

### L2 — 多 Agent 对抗循环（所有子任务完成后）

L2 是一个**审查者 ↔ 执行者收敛循环**，不是一次性审查。

```
┌─ 审查者 Agent (sonnet) ◄──────────────────────┐
│  审查代码，输出 ISSUES [...] 或 PASS              │
├─ PASS? ─── 是 → 进入 L3 ──────────────────────→ 退出
│         └─ 否 ↓                                  │
│  收敛检测:                                        │
│    blocker 数 ≥ 上轮? → 停止，升级给用户 ────────→ 退出
│    同一 blocker 连续 2 轮? → 标记需人工，移除      │
│    已达 3 轮? → 升级给用户 ──────────────────────→ 退出
│         ↓ (收敛中，继续)                          │
│  执行者 Agent (sonnet)                           │
│  修复问题，git commit                             │
│                                                  │
└─ 回到审查者 ──────────────────────────────────────┘
```

**第 1 轮 — 审查者 Agent：**
```
Agent(
  model="sonnet",
  prompt="你是一个严格、独立的代码审查者。

审查以下差异：
{git diff from start of /deep-task}

检查这些维度：
1. 边界情况：边界条件是否已处理？
2. 假设：哪些隐含假设可能出错？
3. 集成：与其他模块有风险吗？
4. 宪法：阅读 .claude/constitution.md，逐条检查合规性
5. 测试：关键路径是否有测试覆盖？

规则：
- 只标记可能导致 Bug 或违反架构的真实问题。不要纠结代码风格。
- 对每个问题，指定：文件、行号、严重性（critical/warning）、问题描述、修复建议。
- 如果一切正常，输出：PASS

输出格式：
ISSUES:
- [critical] file:line — 描述 — 建议修复
- [warning] file:line — 描述 — 建议修复
或：
PASS"
)
```

**如果有 ISSUES → 执行者 Agent：**
```
Agent(
  model="sonnet",
  prompt="修复代码审查发现的以下问题。
不要与审查者争论 — 直接修复每个问题。
修复后运行相关测试确认。
git commit 你的修复。

需要修复的问题：
{审查者的问题列表}

规则：
- 修改前阅读 .claude/constitution.md
- 每个问题或一组相关问题一次提交
- 修复后运行测试"
)
```

**然后回到审查者**，使用更新后的差异。重复直到 PASS 或达到收敛上限。

**收敛检测**（替代固定 3 轮限制）：

每轮审查者结束后，计算 blocker（critical 问题）数。应用以下规则：
- **PASS** → 退出进入 L3
- **Blocker 数比上轮减少** → 继续（正在取得进展）
- **Blocker 数不变或增加** → **立即停止**，通过 `AskUserQuestion` 升级给用户，附带剩余问题。继续只会浪费预算而无进展。
- **同一 blocker 连续出现 2 轮**（执行者未能修复）→ 标记为"需人工介入"，从执行者列表中移除，继续处理剩余问题
- **硬上限：3 轮** — 即使在收敛中，3 轮后也停止，升级所有剩余问题

### L2-Alt — 测试对抗循环（用于关键路径）

对标记为关键的子任务，运行额外的对抗模式：

```
┌─ 测试者 Agent ◄───────────────────┐
│  写边界测试和破坏性测试              │
│  输出: 新测试文件                   │
├─ 测试全过? ── 是 → 完成 ─────────→ 退出
│           └─ 否 ↓                  │
│  执行者 Agent                      │
│  让失败的测试通过（不能删测试）      │
│  git commit                        │
│                                    │
└─ 回到测试者，写更多边界测试 ─────────┘
   (最多 3 轮)
```

**测试者 Agent：**
```
Agent(
  model="sonnet",
  prompt="你是一名 QA 工程师，目标是打破这段代码。

阅读差异：{git diff}
阅读验收标准：{原始目标标准}

编写测试：
1. 测试边界值（0、-1、MAX、空、null）
2. 测试并发/竞态条件（如适用）
3. 测试错误路径（网络故障、无效输入、磁盘满）
4. 测试开发者可能没想到的组合

不要写正向路径测试 — 那些已经存在。
输出：带描述性名称的测试文件。运行它们。报告哪些通过哪些失败。"
)
```

### L3 — 端到端（最终关卡）
1. 运行完整测试套件（包括 L2-Alt 的所有测试）
2. 检查原始目标的验收标准
3. 宪法合规审计（逐条检查）
4. 如果全部通过 → 进入阶段 5
5. 如果失败 → 修复 → 从 L2 重新验证

## 阶段 5：元学习

分析执行轨迹并保存洞察：

1. 实际复杂度 vs. 估计的？（S/M/L/XL 分类是否正确？）
2. 哪些策略有效？哪些失败了？
3. 有值得记住的新模式吗？
4. 时间/工作量估计有多准确？

### 5.0 成本报告

所有执行完成后，向用户输出成本摘要：

```
## 成本报告

| 类别 | 次数 | 估计 Tokens (输入/输出) | 估计成本 |
|------|------|------------------------|----------|
| 阶段 3 Agent | {N} 次调用 | {input}K / {output}K | ${amount} |
| L2 审查轮次 | {N} 轮 | {input}K / {output}K | ${amount} |
| L2-Alt 测试轮次 | {N} 轮 | {input}K / {output}K | ${amount} |
| **合计** | | | **${total}** |

模型分布：haiku × {N}，sonnet × {N}，主上下文 × 1
```

估算规则：
- 每次 Agent 调用 ≈ 8K 输入 + 2K 输出 tokens（基线）
- 加上文件内容 tokens：读取行数 × 4 tokens/行
- 定价：haiku $0.25/$1.25 per M，sonnet $3/$15 per M，opus $15/$75 per M
- 这是**粗略估算** — 明确标注为近似值

同时将成本追加到学习条目（阶段 5.1）中：
```
- **成本**: ~${total}（{N} 次 agent 调用：haiku × {N}，sonnet × {N}）
```

### 5.1 写入项目：`.claude-flow/learnings/{domain}.md`

学习记录按领域拆分，防止文件膨胀并支持定向检索。

**领域分类**：从主要受影响的模块/领域推导（如 `auth`、`api`、`data-model`、`ci`、`perf`、`ui`）。使用 kebab-case。不确定时用 `general`。

**文件结构**：每个领域文件有 YAML 头和按时间排列的条目：

```markdown
---
domain: {domain}
entry_count: {N}
last_pruned: {date}
---

### {date} — {目标摘要} [score: {1-5}]

- **复杂度**：估计 {X}，实际 {Y}
- **有效策略**：{列表}
- **失败策略**：{列表及原因}
- **发现的陷阱**：{列表}
- **验证备注**：L{N} {足够/不足}，因为 {原因}
- **成本**：~${total}（{N} 次 agent 调用）
```

**相关性评分**（1-5）：此学习对同领域未来任务的适用概率。
- 5：通用陷阱（如"在此模块中永远不要做 X"）
- 4：可复用策略模式
- 3：对类似任务有用的上下文
- 2：一次性洞察，复用性低
- 1：边缘性，清理候选

**索引文件**：维护 `.claude-flow/learnings/INDEX.md`，列出所有领域文件及条目数：
```markdown
- auth.md（3 条，最近：2026-03-15）
- api.md（5 条，最近：2026-03-20）
```

### 5.2 写入 Claude 记忆：`memory/meta_{domain}.md`

这是 **Claude 的私有记忆**，用于跨项目模式匹配。写入或更新：

```markdown
---
name: meta_{domain}
description: {domain} 任务的元策略
type: feedback
---

## {Domain} 任务

### 最佳策略
- {洞察}（验证于：{date}）
```

## 禁止操作

- 不要跳过阶段 0 复杂度分类
- 不要将 L/XL 任务走 S/M 快速路径
- 不要覆盖基于复杂度的模型路由（如对 C:1 任务使用 sonnet 或对 C:4 任务使用 haiku）
- 不要对涉及核心逻辑的 L/XL 任务跳过 L2 验证（使用验证级别选择器来判断豁免）
- 不要在未通过 L1 检查时提交
- 不要在 XL 分解未经用户确认时继续
- 不要在 L/XL 任务的 L3 验证前宣布完成
- 不要重复失败策略超过两次
